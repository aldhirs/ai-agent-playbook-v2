GATE 3

Planning & Task Breakdown

Dari tech design signed ke daftar task yang Dev bisa claim. Target: 1-2 jam. Fokus: granularity konsisten, dependency jelas, paralelisasi optimal, estimate berbasis T-shirt (bukan jam).

Detailed Flow

  1. Run /plan <feature-slug> setelah TD signed.
  2. Planner Agent baca TD & existing code. Identifikasi: file yang harus diubah, file baru, endpoint baru, migration, observability hook, test scaffold.
  3. Generate G3-IMPL-PLAN.md dengan topological task list. Setiap task: ID, judul, estimate (T-shirt XS/S/M/L), dependency, parallelizable y/n, risk flag (red/yellow/green).
  4. Dependency graph di-render Mermaid. Critical path diidentifikasi eksplisit.
  5. Wajib include task pertama: "Setup feature flag + folder evidence/". Task terakhir: "Cleanup feature flag + post-launch retro draft". Paralel: "Tester Agent generate test plan" dimulai bersamaan dengan task implementasi pertama.
  6. Tech Lead validate breakdown. Cek granularity: 1 task = 1 PR yang bisa di-review <30 menit. Kalau ada task L, paksa pecah.
  7. Tech Lead validate risk flag. Task red butuh spike sebelum mulai. Task yellow butuh pair atau extra review.
  8. Dev claim task (di Jira / Linear). Auto-assignment kalau MCP terhubung.
  9. Gate 3: Tech Lead merge plan PR. Otomatis create task di tracker (kalau MCP aktif). Feature status → "In Development".

Cara Kerja AI & Cara Kerja Human

🤖 Planner Agent

  • Baca G2-TECH-DESIGN.md + existing code untuk identifikasi file yang berubah
  • Generate task list urut topologis dengan dependency graph (Mermaid)
  • Estimate T-shirt (XS/S/M/L) — bukan jam (jam selalu salah)
  • Risk flag: red (perlu spike), yellow (perlu pair), green (straightforward)
  • Tag suggestion untuk auto-assign di tracker: backend/frontend/db-migration/contract/infra/test
  • Identifikasi paralelisasi maksimum (test plan paralel dengan dev — kunci pemangkasan waktu)

Yang TIDAK boleh:

  • Estimate dalam jam
  • Task ukuran >L (paksa pecah)
  • Task tanpa owner role yang jelas
  • Lupa task feature flag setup + cleanup

Prompt: commands/plan.md

👤 Tech Lead + Developer

Tech Lead:

  • Validate granularity (1 task = 1 PR <30 menit review)
  • Validate dependency graph; cek critical path realistis
  • Identifikasi task red yang butuh spike → assign ke senior
  • Setujui pair-up untuk task yellow

Developer:

  • Claim task sesuai kapasitas & skill
  • Kalau merasa task terlalu besar, propose pecahan ke Tech Lead
  • Cek dependency sebelum mulai (task yang block harus closed dulu)

Aturan Granularity Task

1 task = 1 PR yang bisa di-review <30 menit.
  • XS (½ - 1 jam): config change, dependency bump, copy edit
  • S (½ - 1 hari): single function, satu endpoint, satu komponen UI sederhana
  • M (1-2 hari): feature kecil, multi-file tapi 1 layer (handler atau usecase atau repo)
  • L (2-3 hari): MAX. Lebih dari ini wajib pecah jadi M atau S
Estimate >L = sinyal task belum cukup dipikirkan. Pecah dulu.

Anti-pattern Khusus Gate 3

  1. Estimate dalam jam. Jam selalu meleset. T-shirt cukup untuk planning + di akhir kuartal velocity bisa di-kalibrasi.
  2. "Bikin endpoint" sebagai 1 task. Pecah jadi: contract, handler, validation, observability, test, evidence.
  3. Tidak ada task feature flag. Setup flag harus task pertama; cleanup harus task terakhir.
  4. Test plan dimulai setelah dev selesai. Test plan harus paralel — dimulai bersamaan task implementasi pertama.
  5. Task red tanpa spike. Task red & mulai langsung = kemungkinan besar miss-estimate & rework.
  6. Dependency graph tidak di-render. Tanpa visual, paralelisasi pasti suboptimal.

Kalau Macet di Gate Ini

Task terlalu banyak ber-flag merah (red). Sinyal: TD belum cukup detail di area tertentu. Aksi: balik ke Gate 2, expand section yang relevan, rerun /plan.
Dev tidak yakin estimate. Wajar untuk task baru. Aksi: tag yellow + pair-up; jangan dipaksa green.
