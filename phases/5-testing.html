GATE 5

Testing

Test plan disiapkan paralel dengan development (mulai bersamaan Gate 3, bukan setelah Gate 4). Ini kunci kenapa testing tidak lagi blocker. Fokus: matrix state × action, edge case yang AI-generated lalu QA tambahkan domain-specific, automation strategy per layer.

Human Owner
QA Engineer
AI Agent
Tester
Input
PRD + TD (AC + failure modes)

TEST-PLAN.md + automated tests

Pass rate ≥95% · QA tag qa-greenlight
Mulai Kapan
Bersamaan Gate 3 (paralel)

Detailed Flow

  1. QA run /test-plan <feature-slug> setelah TD signed (Gate 2) — tidak menunggu dev selesai.
  2. Tester Agent baca: PRD (AC sebagai sumber utama), TD (failure modes, edge cases), banned-field test existing, pattern test di service standard layout (test/integration/, test/e2e/) untuk konsistensi style.
  3. Generate TEST-PLAN.md: mapping AC → test case, test matrix state × action (wajib 5 state: default/loading/empty/error/success), 10+ edge case, automation strategy per layer, regression set.
  4. Generate skeleton test file di features/<slug>/tests/integration/ & features/<slug>/tests/e2e/ — struktur describe/it lengkap, assertion belum diisi. Tooling test (Go testcontainers, Playwright) di-config baca dari path feature folder ini.
  5. QA review & tambahkan domain-specific. Yang AI sering miss: regulasi (PCI, GDPR), quirks integration partner, tribal knowledge soal user behavior.
  6. QA isi assertion di skeleton test (atau pair dengan Dev kalau test complex). Test plan = drafting Anchor; assertion implementasinya manual.
  7. QA execute test begitu Dev merge task yang relevan. Tidak menunggu semua task selesai — test berjalan inkremental.
  8. Tester bantu triage failure (kategori: bug code · bug test · flaky · environment).
  9. Gate 5: qa-greenlight. Exit criteria: 100% TC executed, pass rate ≥95%, no P0/P1 open, P2 ≤3, banned-field test green, perf budget tidak terlewati. QA tag rilis & sign-off.

Cara Kerja AI & Cara Kerja Human

🤖 Tester Agent

  • Baca PRD AC + TD failure modes sebagai sumber utama
  • Generate test matrix state × action — wajib 5 state
  • Generate 10+ edge case (boundary, format, concurrency, network, auth, data)
  • Automation strategy: tag layer mana (unit/integration/E2E/contract)
  • Generate skeleton test file (describe/it lengkap, assertion empty)
  • Identifikasi regression set (smoke test untuk tiap rilis)
  • Bantu triage saat failure: kategorikan + hipotesis root cause

Yang TIDAK boleh:

  • Generate test yang assertion-nya implementation-specific (mock-heavy, fragile)
  • Skip sad path / edge case (happy-only never enough)
  • Mengasumsikan compliance scenario tanpa input QA

Prompt: commands/test-plan.md

👤 QA Engineer

  • Review & tambahkan edge case domain-specific (regulasi, tribal knowledge, integration quirks)
  • Isi assertion di skeleton test (atau pair dengan Dev)
  • Execute test inkremental — jangan menunggu semua task done
  • Identifikasi flaky test → buka ticket maintain test, jangan biarkan jadi noise
  • Sign-off dengan qa-greenlight hanya kalau exit criteria terpenuhi
  • Untuk feature high-risk (payment/auth/data deletion) — perketat exit criteria (P2 ≤1)

Yang TIDAK boleh:

  • Tunggu sampai dev selesai semua baru mulai test plan
  • Sign-off dengan defect P1 open
  • Manual test scenario yang sebenarnya bisa diautomasi
  • Skip exploratory testing (45 menit per rilis wajib)

Edge Cases yang Selalu Dicek

  • Boundary: input minimum/maximum (panjang, nilai, jumlah item)
  • Format: locale, timezone, encoding (UTF-8, emoji), float precision
  • Concurrency: 2 request bersamaan dengan Idempotency-Key sama
  • Network: slow 3G, intermittent, offline, timeout di tengah request
  • Auth: token expired, scope tidak cukup, cross-tenant access
  • Data: NULL, empty string, very long string, special char, PII shape
  • Compliance: banned-field test (PII tidak muncul di log)

Exit Criteria (Gate 5)

QA bisa tag qa-greenlight hanya kalau (semuanya):
  1. 100% test case di matrix executed
  2. Pass rate ≥ 95%
  3. Zero P0/P1 defect open
  4. P2 defect ≤ 3
  5. Defect density ≤ <X bug per 100 LOC changed> (set per service)
  6. Banned-field test green
  7. Performance budget tidak terlewati
  8. Untuk feature high-risk: P2 ≤ 1, performance budget ketat (≤90% threshold)

Anti-pattern Khusus Gate 5

  1. Test plan dimulai setelah dev selesai. Itu polanya v1 — yang menyebabkan testing jadi blocker. Test plan WAJIB mulai bersamaan Gate 3.
  2. Happy path-only. Sad path & edge case adalah inti. Happy-only never enough.
  3. Test E2E untuk semua hal. E2E mahal & lambat. Pakai untuk happy + sad utama; sisanya integration / unit.
  4. Flaky test dibiarkan. Sekali flaky, harus diperbaiki atau di-quarantine. Flaky merusak trust ke suite.
  5. Skip exploratory testing. 45 menit eksplorasi per rilis sering nemu bug yang test automated miss.
  6. Manual test yang seharusnya bisa diautomasi. Ulang >3× = wajib auto.

Kalau Macet di Gate Ini

Defect terlalu banyak (P0/P1 tidak turun setelah 2 putaran fix). Sinyal: ada masalah arsitektur, bukan test issue. Aksi: stop, eskalasi ke Tech Lead, mungkin balik ke Gate 2 untuk re-think.
Flaky test merusak signal. Aksi: quarantine flaky test, buka ticket investigation. Jangan dipaksa pass dengan retry liar.
