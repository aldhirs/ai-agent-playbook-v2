GATE 4

Development

Dari task assigned ke PR merged dengan evidence. Target: 40-60% lebih cepat dari baseline (per task). Fokus: Dev tetap accountable; AI generate boilerplate, scaffold test, draft evidence — bukan "AI bikin semuanya, dev tinggal merge".

Developer
AI Agent
Coder + Reviewer
Input

40-60% per task

Detailed Flow (per task)

  1. Dev cek prasyarat: task tidak punya dependency yang belum closed. Baca task spec di IMPL-PLAN.
  2. Dev run /implement <feature-slug>/<task-id>. Coder Agent baca CLAUDE.md hierarchy, PRD, TECH-DESIGN, existing code untuk consistency.
  3. Coder implement. Pakai pattern service (layout cmd/internal/usecase/handler/repository untuk Go; composition API + Pinia + 5 UI state untuk Vue). Update features/<slug>/contract/openapi.yaml atau features/<slug>/contract/proto/ sesuai TD. Tulis unit test minimal happy + 2 sad path. Tambahkan file code yang dimodifikasi ke features/<slug>/code-manifest.md.
  4. Coder generate evidence di features/<slug>/evidence/<task-id>/: curl.txt (request/response), test.log (output go test -v), screenshots/ kalau UI change (min. 3: default, loading, error).
  5. Dev review output Coder. Validate: logika benar? edge case missed? error message friendly? Bagian yang Coder tidak yakin akan disebut eksplisit — Dev wajib pelajari ulang.
  6. Dev iterate dengan Coder sampai puas (biasanya 1-3 putaran).
  7. Dev run /code-review <feature-slug>/<task-id>. Reviewer Agent jalankan self-review checklist (contract conformance, observability, privacy, idempotency, evidence completeness) + static analysis summary + AC coverage check.
  8. Dev fix red findings dari Reviewer. Hijau / kuning yang reasonable bisa di-defer; merah wajib fix.
  9. Dev run make verify-feature FEATURE=<slug>. Harus exit 0. Tanpa ini, PR jangan dibuka.
  10. Dev open PR. Title sebut task ID. Description copy self-review checklist + AC coverage table dari Reviewer.
  11. Peer review fokus ke keputusan (logic, contract, observability) — bukan style (sudah dihandle linter). SLA 4 jam kerja.
  12. Gate 4: PR merged. CI hijau + peer approve + evidence folder tidak kosong. Lanjut ke Gate 5 (parallel sebenarnya: testing sudah jalan).

Cara Kerja AI & Cara Kerja Human

🤖 Coder + Reviewer Agent

Coder Agent:

  • Baca CLAUDE.md hierarchy (org → tribe → service) — wajib semuanya
  • Baca PRD + TD untuk AC & design decisions
  • Implement dengan pattern existing service (jangan reinvent)
  • Generate unit test minimal happy + 2 sad
  • Generate evidence: curl.txt, test.log, screenshots/
  • Self-review checklist di PR description
  • Jangan modifikasi file di luar scope task; kalau perlu touch → warning ke user

Reviewer Agent:

  • Run checklist 6 kategori: contract, code quality, observability, privacy/security, test, evidence
  • Static analysis summary (top 3 finding)
  • AC coverage table — setiap AC di PRD → test mana yang verify
  • Output rekomendasi: ready / fix-first (dengan list spesifik)

Yang TIDAK boleh:

  • Coder over-claim ("semua sudah selesai") — wajib sebut bagian yang butuh review manual
  • Reviewer nag soal style (itu noise — linter sudah handle)
  • Klaim evidence tanpa cek folder benar-benar berisi 3 file wajib

Prompt: commands/implement.md & commands/code-review.md

👤 Developer + Peer Reviewer

Developer accountable untuk:

  • Validate logika output Coder (bukan rubber-stamp)
  • Tanggung jawab penuh untuk code yang merged (Coder = alat bantu, bukan owner)
  • Fix red findings dari Reviewer Agent
  • Pastikan make verify-feature exit 0
  • Self-review checklist di PR description

Peer reviewer accountable untuk:

  • Fokus ke keputusan, bukan style
  • Tanya: "Apakah error case ini handled?"; "Idempotency-Key sudah?"; "Observability cukup?"
  • SLA 4 jam kerja; di luar itu, auto-ping

Yang TIDAK boleh:

  • Merge PR tanpa evidence folder
  • "AI bilang udah benar" sebagai justifikasi
  • Skip make verify-feature

Definition of Done untuk 1 Task

Task = DONE kalau (semuanya):
  1. Code diff sesuai task scope · tidak menyentuh file di luar scope tanpa diskusi
  2. Unit test pass · minimal happy + 2 sad
  3. Evidence folder ada 3 file wajib (curl.txt, test.log, screenshots/ kalau UI)
  4. make verify-feature FEATURE=<slug> exit 0
  5. CI hijau di PR
  6. Reviewer Agent self-review: tidak ada red finding
  7. Peer review approve
  8. Self-review checklist (7 item) di PR description
Tanpa salah satu di atas = TIDAK done, terlepas dari "kayaknya jalan".

Anti-pattern Khusus Gate 4

  1. "AI bilang udah benar." Dev tetap accountable. Kalau bug di prod, "AI bilang" bukan jawaban.
  2. Evidence kosong / placeholder. Tanpa evidence = belum done. Tidak ada exception.
  3. Skip make verify-feature. Itu satu-satunya cara cek otomatis bahwa kontrak masih utuh.
  4. Modifikasi file di luar task scope. Kalau perlu refactor lain, buat task baru — jangan smuggling di PR yang seharusnya fokus.
  5. Peer review <5 menit untuk PR ukuran M. Sinyal review tidak serius. Counter: ukur median review time per ukuran task.
  6. Unit test cuma happy path. Minimal 2 sad path wajib. Test name informatif.
  7. PR description kosong / cuma "see task". Wajib checklist + AC coverage table.

Kalau Macet di Gate Ini

Coder generate code yang aneh / tidak sesuai pattern service. Sinyal: CLAUDE.md service belum lengkap (tidak punya cukup konteks). Aksi: update CLAUDE.md service, rerun /implement.
Test gagal karena flaky. Tidak boleh di-merge. Aksi: investigate root cause; biasanya share state antar test atau race. Catat sebagai lesson.
Dev menemukan kontrak TD ternyata salah. Aksi: balik ke Gate 2, update TD, re-LGTM. Memang menyebalkan, tapi merge dengan TD yang salah = utang besar.
