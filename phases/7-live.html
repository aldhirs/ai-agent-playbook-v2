GATE 7

Live & Post-launch

Rilis ke produksi dengan runbook, observability, & rollout plan. Post-launch (T+7 hari) retro otomatis untuk extract lesson. Tujuan: tidak ada "launched and forgotten" — setiap rilis punya feedback loop yang structured.

Detailed Flow

  1. Run /runbook <feature-slug> (atau ke Reliability Agent). Baca TD (observability plan, failure modes, rollback) + alert config existing.
  2. Generate runbook: alur on-call response untuk top 3-5 failure mode. Tiap entry: symptom → first-check → mitigation → escalation path.
  3. Generate alert config sesuai observability plan TD (Grafana / Datadog YAML). SLO target ditulis eksplisit.
  4. Generate dashboard panel baru di folder team. Panel: traffic, latency p50/p99, error rate, custom metric dari feature.
  5. SRE validate runbook. Cek: bisakah on-call junior ikuti tanpa context tambahan? Mitigation reversible?
  6. Rilis dengan canary: feature flag on 1% traffic for 24h → 10% for 24h → 100%. Default. Big-bang hanya dengan alasan eksplisit di TD.
  7. SRE / Tech Lead monitor canary: SLO terpenuhi? error rate naik? complaint user? Kalau ada anomaly → rollback (toggle flag off), bukan "wait and see".
  8. T+7 hari otomatis: Reliability Agent generate post-launch retro draft dari log + metric + incident yang muncul.
  9. Tim review retro di 30 menit meeting (max). Output: 1-3 lesson baru ditulis di tribes/<id>/LESSONS.md.
  10. Gate 7: SRE checklist hijau + retro file committed. Feature flag cleanup task di-claim. Feature status → "Live & Maintained".

Cara Kerja AI & Cara Kerja Human

🤖 Reliability Agent

  • Baca TD observability plan + failure modes
  • Generate runbook untuk top 3-5 failure mode
  • Generate alert config YAML (Grafana / Datadog)
  • Generate dashboard panel definition
  • T+7: scrape log + metric + incident, generate retro draft
  • Compare aktual vs SLO target di TD; flag deviasi

Yang TIDAK boleh:

  • Auto-deploy ke produksi (selalu human-triggered)
  • Generate runbook tanpa mitigation step
  • Skip mention rollback path

👤 SRE / Tech Lead

  • Validate runbook bisa diikuti on-call junior
  • Monitor canary stage demi stage
  • Decide promotion ke stage berikutnya (1% → 10% → 100%)
  • Decide rollback kalau ada anomaly (toggle flag, bukan tunggu)
  • Facilitate post-launch retro
  • Sign-off SRE checklist sebelum closure

Yang TIDAK boleh:

  • "Wait and see" saat ada anomaly di canary (selalu rollback dulu, debug kemudian)
  • Skip post-launch retro
  • Tidak cleanup feature flag (utang technical)

SRE Checklist Pre-Release

Sebelum canary 1% start, semua wajib hijau:
  1. Runbook lengkap & validated
  2. Alert config deployed (yang relevant: error rate spike, latency p99 breach, custom feature failure)
  3. Dashboard panel live
  4. Feature flag ada & default off di produksi
  5. Rollback path tested (toggle flag → bisa balik dalam <5 menit)
  6. Migration backward-compatible (no destructive change yang tidak bisa di-rollback)
  7. On-call notified (Slack + tag di rotation tool)
  8. Release note ditulis (untuk customer-facing change)

Post-launch Retro (T+7 Hari)

Format retro otomatis (draft oleh AI, refine oleh tim):
  1. Aktual vs Plan: lead time gate 1→7 aktual, defect escape rate, perf metric aktual vs SLO
  2. Incident (kalau ada): apa terjadi, MTTR, root cause
  3. What worked: 2-3 hal yang efektif (untuk replikasi)
  4. What didn't: 2-3 hal yang nyangkut (untuk improvement)
  5. Lessons: 1-3 lesson baru ditulis ke LESSONS.md
  6. Action items: owner + deadline
Max 30 menit meeting. Kalau lebih lama, fokus retro hilang.

Anti-pattern Khusus Gate 7

  1. Big-bang rilis ("toh sudah QA-green"). Default selalu canary. Real users selalu menemukan sesuatu yang QA miss.
  2. "Wait and see" saat anomaly di canary. Rollback first, debug second. Rollback <5 menit harus reflexive.
  3. Runbook tanpa mitigation step. Runbook yang cuma "page Tech Lead" bukan runbook — itu escalation tree.
  4. Skip post-launch retro. Retro adalah cara extracting lesson. Skip = lesson loop mati = playbook stagnan.
  5. Feature flag tidak di-cleanup. Flag yang tetap ada setelah feature stable = utang technical & sumber bug masa depan.
  6. Alert generic "service down". Alert wajib aksi-driven; on-call harus tahu apa yang harus dia lakukan dalam 30 detik baca.

Kalau Macet di Gate Ini

Canary error rate naik tapi tidak meyakinkan rollback. Default tetap: rollback first. "Investigate live" cuma bila ada SLO budget cukup. Sebagai aturan: kalau ragu, rollback. Re-deploy <1 jam.
Migration tidak bisa di-rollback (sudah destructive). Sinyal: TD/Gate 2 ada kesalahan. Lesson WAJIB ditulis. Untuk recovery: forward-fix dengan migration baru, bukan rollback.
