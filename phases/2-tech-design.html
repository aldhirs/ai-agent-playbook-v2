GATE 2

Tech Discussion

Dari PRD signed ke tech design + ADR + API contract yang lengkap. Target: 4-6 jam (sebelumnya 2-3 hari). Fokus: tiga opsi minimal, pilih dengan alasan eksplisit, kontrak API tegas, rollout reversible.

Input

Target Durasi
4-6 jam

Detailed Flow

  1. Tech Lead validate prasyarat: PRD harus berstatus SIGNED. Cek di features/<slug>/PRD.md.
  2. Tech Lead run /tech-design <feature-slug>. Architect Agent membaca PRD, CLAUDE.md hierarchy, OpenAPI/proto existing, dan service map.
  3. Architect generate minimal 3 opsi. Wajib: 1 opsi konservatif (paling sedikit mengubah arsitektur), 1 opsi agresif (membuka kapabilitas baru), 1 di tengah. Tiap opsi: cara kerja, pros, cons, effort (S/M/L), risk (L/M/H).
  4. Trade-off matrix. Bandingkan 5 aspek minimum: latency, effort, operational complexity, regression risk, reversibility.
  5. Architect drafting: Decision (= ADR inline) + rationale yang merujuk eksplisit ke matrix, API contract (OpenAPI snippet untuk REST atau proto untuk gRPC), data model + migration plan (kalau ada DDL), sequence diagram (Mermaid), failure modes, observability plan, rollout plan.
  6. Tech Lead review & pilih opsi. Tidak boleh "AI bilang option B"; Tech Lead harus bisa jelaskan kenapa B menang dari A & C.
  7. Tag cross-tribe reviewer. Wajib minimal 1 reviewer dari tribe lain (dari daftar buddy di tribes/<id>/CLAUDE.md). Tujuan: cross-pollination, bukan permission.
  8. Cross-tribe reviewer kasih ≥1 pertanyaan/saran spesifik (≤24 jam SLA; lewat itu auto-skip — tidak block). "LGTM" kosong tidak diterima.
  9. Tech Lead iterate kalau perlu (biasanya 0-1 putaran kecil). Open Items di TD harus tuntas; maks 5, kalau lebih = design belum matang.
  10. Gate 2: LGTM-TD ×2. Tech Lead + cross-tribe reviewer tag LGTM-TD di TECH-DESIGN.md. Status → SIGNED. Lanjut ke Gate 3.

Cara Kerja AI & Cara Kerja Human

🤖 Architect Agent

Yang dilakukan:

  • Baca PRD + CLAUDE.md hierarchy + existing OpenAPI/proto untuk konsistensi naming
  • Generate minimal 3 opsi termasuk satu yang konservatif
  • Trade-off matrix dengan 5 aspek wajib
  • Mermaid sequence diagram (bukan ASCII)
  • API contract dengan validation matrix & 422 friendly message
  • Migration plan: backward-compatible only (dual-write 1 release sebelum drop column)
  • Rollout default: feature flag + canary 1% → 10% → 100%, rollback reversible

Yang TIDAK boleh dilakukan:

  • Loncat langsung ke opsi "paling keren" tanpa bandingkan 3
  • Ubah contract endpoint existing tanpa migration path
  • Skip section template
  • Bikin Decision tanpa rationale berbasis matrix (tidak boleh: "karena best practice")

Prompt yang dipakai: commands/tech-design.md

👤 Tech Lead & Cross-tribe Reviewer

Tech Lead accountable untuk:

  • Pilih opsi (tidak boleh delegasikan ke AI) & tulis konsekuensi yang diterima
  • Validate API contract: backward-compatible? naming sesuai konvensi tribe?
  • Validate observability plan: log/metric/trace cukup untuk debug pos-launch?
  • Validate rollout: reversible? feature flag namanya jelas?
  • Tag cross-tribe reviewer yang relevan

Cross-tribe reviewer accountable untuk:

  • Tidak hanya "LGTM" — wajib ≥1 pertanyaan/saran spesifik (misal: "Service kami juga handle idempotency seperti ini, sudah cek konsistensi?")
  • Bawa pengalaman dari tribe-nya: pattern yang berhasil, kegagalan yang pernah dialami
  • ≤24 jam SLA; setelah itu auto-skip tanpa block

Yang TIDAK boleh dilakukan:

  • "AI bilang option B" tanpa rationale dari Tech Lead
  • Skip cross-tribe review (kecuali emergency lane)
  • Approve TD dengan Open Items >5

Approval Mekanisme

Gate 2 ditutup oleh:
  1. LGTM-TD dari Tech Lead di file TECH-DESIGN.md.
  2. LGTM-TD dari cross-tribe reviewer (dari daftar buddy di CLAUDE.md tribe).
  3. CI check: file ada, semua section template terisi, mermaid syntax valid, OpenAPI/proto lint pass.
  4. PR merged → status feature → "Ready for Planning".

Anti-pattern Khusus Gate 2

  1. Tek Lead langsung pilih opsi AI tanpa bandingkan. AI proposes, human approves — bukan "human rubber-stamps".
  2. Cuma 1 opsi di TD. Tanpa opsi alternatif, kita tidak bisa mengukur trade-off. Min. 3 wajib.
  3. Cross-tribe reviewer pasif. Reviewer cuma jadi formalitas. Counter: ukur jumlah pertanyaan/saran rata-rata per review; kalau <1, sinyal mati rasa.
  4. Contract breaking change tanpa migration. Endpoint existing yang dipakai client lain — tidak boleh diubah tanpa dual-write / versioning.
  5. Rollout big-bang. Default selalu canary. Big-bang hanya kalau benar-benar tidak bisa di-feature-flag, dan harus disebut alasannya.
  6. Migration destructive di rilis pertama. DROP COLUMN di rilis yang sama dengan code change = rollback mati. Selalu pisah 2 rilis.

Kalau Macet di Gate Ini

3 opsi semuanya sama-sama tidak meyakinkan. Sinyal: PRD belum cukup tajam, atau ada constraint yang belum ditangkap. Aksi: balik ke PM, tanya constraint apa yang belum jelas (regulasi? performa? budget?), update PRD, rerun /tech-design.
Cross-tribe reviewer tidak bisa di-find / nggak ada yang relevan. Aksi: skip dengan eksplisit (catat di TD) dan eskalasi ke forum tech lead untuk diskusi 15 menit. Jangan dipaksa lulus tanpa peer review.
Setelah TD signed, Dev ternyata menemukan asumsi yang salah. Aksi: balik ke Gate 2, jangan diam-diam ubah TD. Update TD, re-LGTM. Memang menyebalkan, tapi shortcut di sini akan membayar 5× di gate berikutnya.
