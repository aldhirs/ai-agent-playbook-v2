GATE 6

Defects & QA Greenlight

Triage bug dari Gate 5 dengan disiplin: klasifikasi prioritas, root-cause hypothesis, lesson candidate untuk pattern recurring. Tujuan bukan hanya menutup bug — tapi memastikan bug tidak terulang dengan lesson loop.

Detailed Flow per Defect

  1. QA log defect di tracker dengan template: judul, repro step, expected vs actual, environment, attachment (screenshot/log/curl).
  2. Triage Agent classify otomatis: prioritas (P0/P1/P2/P3), area (backend/frontend/contract/infra), root-cause hypothesis (3 kandidat dengan likelihood).
  3. Cek apakah ini lesson candidate. Triage cocokkan dengan LESSONS.md existing — kalau pattern serupa pernah muncul, flag sebagai recurring.
  4. Tech Lead validate klasifikasi. Adjust prioritas kalau perlu (Triage konservatif untuk under-estimate). Assign owner fix.
  5. Dev investigate & fix. Untuk P0/P1, pair dengan reviewer. Fix wajib disertai test yang reproduce bug (regression test).
  6. QA verify fix. Jalankan test plan + skenario yang relevan + exploratory.
  7. Kalau pattern recurring: tulis lesson di tribes/<id>/LESSONS.md dengan format konsisten (run /lesson "<judul>").
  8. Gate 6: QA sign-off. Semua P0/P1 closed, P2 ≤3, lesson recurring sudah di-tulis. Status feature → "Ready for Release".

Cara Kerja AI & Cara Kerja Human

🤖 Triage Agent

  • Auto-classify prioritas berdasarkan: severity user impact, breadth (berapa banyak user terkena), workaround availability
  • Identifikasi area (backend, frontend, contract, infra, third-party)
  • Generate 3 root-cause hypothesis dengan likelihood (high/medium/low)
  • Cocokkan dengan LESSONS.md — kalau ada pattern, flag recurring
  • Kalau bug crosses tribe boundary, tag tribe terkait
  • Generate regression test skeleton (yang harus diisi assertion oleh dev)

Yang TIDAK boleh:

  • Auto-close defect berdasarkan classification saja
  • Down-rank prioritas (konservatif: kalau ragu, naikkan)

👤 QA + Tech Lead + Dev

QA:

  • Log defect dengan reproducibility yang clear (siapa pun harus bisa reproduce)
  • Verify fix (bukan cuma re-test, juga regression set)
  • Tag lesson kalau pattern recurring

Tech Lead:

  • Validate klasifikasi prioritas Triage
  • Assign owner fix (sesuai kapasitas)
  • Approve fix sebelum merge
  • Decide lesson promotion (tribe → org kalau muncul lintas tribe)

Dev:

  • Reproduce dulu sebelum fix (kalau tidak bisa reproduce, eskalasi balik ke QA)
  • Fix wajib disertai regression test
  • Self-review checklist (Gate 4) berlaku juga untuk fix PR

Format Lesson (Wajib)

## [LESSON-yyyy-mm-dd-N] <Judul singkat>

**Context:** dimana & kapan ini muncul

**Root cause:** akar masalahnya (bukan symptom)

**Prevention:**
- [ ] Aksi konkret 1 (mis. tambah test type X)
- [ ] Aksi konkret 2 (mis. update CLAUDE.md service)

**Related:** principle CLAUDE.md §X.Y · related lessons LESSON-yyyy-mm-dd-M

**Status:** active / promoted-to-tribe / promoted-to-org / retired

Lesson Promotion (Cross-Tribe)

Pattern yang muncul di ≥2 service di tribe sama → promote ke tribes/<id>/CLAUDE.md sebagai do/don't. Pattern di ≥2 tribe → promote ke org/.claude/CLAUDE.md sebagai principle baru. Triage Agent scan otomatis tiap bulan.

Anti-pattern Khusus Gate 6

  1. Fix tanpa regression test. Bug yang tidak punya test akan kembali. Selalu reproduce dulu via test, baru fix.
  2. Lesson tidak ditulis untuk pattern recurring. Sinyal lesson loop mati. Owner playbook wajib facilitate retro mini.
  3. QA sign-off dengan P1 open. Tidak boleh, terlepas dari "deadline".
  4. Triage Agent down-rank prioritas tanpa human review. Triage konservatif by default; Tech Lead yang adjust kalau perlu.
  5. "Defect ini bukan tanggung jawab kami" (tribe lain). Bug yang crosses tribe boundary harus di-co-own, bukan dilempar.
  6. Auto-close defect tanpa verify fix. Selalu verify dengan re-test, bukan cuma cek code merged.

Example Walkthrough — Bug Triage

AksiPelaku
QA log bug: "Apple Pay button hang di Safari 17 setelah 5 detik. Repro: open checkout, klik Apple Pay, wait 5s."👤 QA
Triage classify: P1 (core journey, regression dari Safari 17), area frontend. 3 hipotesis: (1) timeout SDK Apple Pay; (2) Safari security policy; (3) iframe sandbox issue. Highest likelihood: timeout (matches LESSON-2026-03-12 di tribe 1)🤖 Triage
Tech Lead confirm P1. Assign Dev B (yang familiar SDK).👤 Tech Lead
Dev B reproduce, find: timeout di SDK call 5s, perlu 10s untuk Safari 17. Fix: bump timeout + tambah retry. Regression test: integration test simulate Safari 17 timing.👤 Dev B
QA verify fix. Pass. Sign-off P1.👤 QA
Pattern matches LESSON-2026-03-12 (Safari quirks). Promote dari tribe-local jadi lesson tribe-wide di tribes/3/LESSONS.md: "Selalu test SDK third-party di Safari latest + Safari latest-1; default timeout 5s sering terlalu pendek."👤 QA + Tech Lead

Kalau Macet di Gate Ini

P1 tidak bisa diperbaiki tepat waktu (vendor third-party). Aksi: turunkan ke P2 + feature flag off untuk Safari 17 saja + commit untuk fix di rilis berikutnya. Sebutkan eksplisit di release note.
Defect tidak reproducible. Aksi: tetap log + tag not-reproducible + minta QA tangkap recording / network trace di session berikutnya. Jangan auto-close.
