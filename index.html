Playbook v2.1 · Team Edition

Cara Tim Bekerja dengan AI Agent

Workflow untuk multi-tribe (1, 2, 3a, 3b, 3c) yang memangkas waktu di fase grooming, tech discussion, development, dan testing — dengan review & approval tetap di tangan developer.

Tagline: Contract first. Evidence always. AI proposes, human approves. Tribe shares, service owns. Analyze existing first. Sync repo & branch.

🎉 What's new in v2.1: Prinsip 19 (Analyze Existing First) + Prinsip 20 (Sync Repo & Branch dari Slug) + Gate 1 approval 3-pihak (PM + Designer + FE Dev) + UI-Impact workflow + Gate-based file naming (G1-PRD.md, dst.) + project-level agents. Lihat README untuk changelog lengkap.

Apa yang Berubah dari v1

Playbook v1 sudah solid sebagai solo-operator setup: prinsip ketat, 12+ agent specialist, evidence gate, dan lessons loop. Tapi saat dipakai tim (4-8 dev × 5 tribe), muncul 4 gap yang harus ditutup:

Gap di v1Solusi di v2
Solo-oriented. Asumsi satu human orchestrasi semua agent. Operating Model dengan role split. 5 human role × 5 AI agent counterpart. Setiap fase punya pemilik manusia + agent pendamping.
Tidak ada handoff antar-human. Konteks hilang saat PM hand-off ke Dev, Dev hand-off ke QA. Feature Folder. Satu folder per feature membawa PRD → tech design → impl plan → test plan → evidence. Dipakai semua peran.
Tidak ada tribe-level governance. Tribe 3a/3b/3c share context tapi v1 tidak punya cara share principle. CLAUDE.md hierarchy 3 layer: org → tribe → service. Tribe 3 punya CLAUDE.md sendiri yang inherit dari org dan di-override di service.
Grooming/PRD & Testing tipis. v1 fokus engineering. Padahal 2 fase ini paling sakit (per Aldhi). Groomer Agent & Tester Agent jadi first-class. Lengkap dengan template PRD, ADR, dan test plan generator.

Yang dipertahankan utuh dari v1: Evidence Gate (3 artifact + make verify-feature), Lessons Loop, dan 14 Principles (CLAUDE.md). Itu sudah benar — kita tinggal tambah lapisan tim di atasnya.

1. Prinsip v2 — 6 Tambahan untuk Tim

v1 punya 14 prinsip teknis. v2 menambahkan 6 prinsip operating + quality khusus untuk tim:

Prinsip 15

AI Proposes, Human Approves

Setiap gate ditutup oleh artefak yang di-approve oleh peran manusia tertentu. Approval = signed comment di PR / tag di doc, bukan verbal. Tanpa approval, gate tidak boleh maju.

Prinsip 16

Tribe Shares, Service Owns

Pola yang dipakai >1 service di tribe yang sama wajib naik ke tribes/X/CLAUDE.md. Pola yang dipakai >1 tribe wajib naik ke org/CLAUDE.md. Konvensi diturunkan, bukan ditemukan ulang.

Prinsip 17

One Folder, All Phases

Setiap feature/epic punya satu folder features/<slug>/ yang berisi seluruh artefak dari grooming sampai post-launch. Tidak ada konteks yang hidup di Slack/DM/kepala orang.

Prinsip 18

Async by Default, Sync on Conflict

Agent menghasilkan draft async (jalan di background). Manusia review batched (pagi/sore). Sync meeting hanya dipanggil saat ada konflik desain, bukan untuk ratifikasi yang AI sudah selesai.

Prinsip 19 ⭐ NEW v2.1

Analyze Existing First, Propose Minimal Change

Sebelum design solusi baru, agent WAJIB audit codebase existing. Tentukan magnitude: NO-CHANGE / EXTEND (default) / NEW-MODULE / REWRITE. Naik ke NEW-MODULE/REWRITE harus justify dengan ADR. Anti-over-engineering by design.

Prinsip 20 ⭐ NEW v2.1

Sync Repo & Branch dari Slug

Agent yang akan modify file di repo target WAJIB git hygiene 3-step: (1) cek working tree clean, (2) git pull --ff-only di base branch, (3) git checkout -b feat/<slug>. Berlaku per-repo kalau feature touch multi-repo.

Kenapa 6 prinsip ini cukup? Mereka menutup 6 mode kegagalan tim: keputusan ambigu (15), reinventing the wheel (16), context drift (17), rapat tak perlu (18), over-engineering (19), dan stale code / branch chaos (20).

2. Operating Model — Human × AI

Setiap fase punya satu pemilik manusia dan satu agent utama. Boleh dipanggil sub-agent lain, tapi yang accountable jelas.

Fase Human Owner AI Counterpart Output
Grooming / Discovery PM / BA Groomer Agent PRD draft, user stories, acceptance criteria, open questions
Tech Discussion Tech Lead Architect Agent Tech design, ADR, OpenAPI/proto contract, sequence diagram, risk register
Planning Tech Lead + Dev Planner Agent Implementation plan, task breakdown dengan estimate & dependency
Development Developer Coder Agent + Reviewer Agent Code, unit test, evidence artifact per task, self-review checklist
Testing QA Engineer Tester Agent Test plan, automated test suite, regression matrix
Defects & Greenlight QA + Tech Lead Triage Agent Defect classification, root-cause analysis, lesson candidate
Live / Post-launch SRE / Tech Lead Reliability Agent Runbook, alert config, dashboard link, post-launch retrospective
Catatan: "Agent" di sini bukan harus jadi entitas teknis terpisah. Praktiknya: 1 Claude Code session + 1 slash command spesifik per fase (lihat Bagian 5). Bisa juga jadi sub-agent terpisah kalau kamu mau strict isolation.

3. Workflow 7 Gate — Dari Ide ke Live

Tiap gate punya input, aksi AI, aksi human, exit criteria, dan artefak. Tidak lulus gate = tidak boleh lanjut. Tidak ada "skip karena buru-buru".

Diagram 1 · Workflow 7 Gate end-to-end. Dev + Test paralel; sisanya sekuensial dengan approval gate eksplisit.
👆 Klik tiap gate untuk lihat detail flow, cara kerja AI & Dev, dan output yang dihasilkan.
1

Grooming & PRD + UI-Impact Pre-Dev (v2.1)

Input: Epic mentah dari Jira/Linear (1-3 paragraf). AI: Groomer Agent baca epic + repo context + lessons + design system catalog (lewat ui-impact-analyst), lalu drafting PRD + UI-IMPACT Pre-Dev (existing page DIFF + reuse plan + 5-state) + open questions. Human: PM jawab open questions, iterate 1-2 putaran. Designer + FE Dev review UI vision. Tech Lead consulted untuk feasibility.

Owner: PM + Designer + FE Dev Agent: Groomer + UI-Impact-Analyst Gate 1: LGTM-PRD ×3 (TL consulted) ~2-4 jam (vs. 2-3 hari)
2

Tech Discussion (+ § 0 Existing Analysis v2.1)

Input: PRD signed (termasuk G1-UI-IMPACT.md Section 1). AI: Architect Agent WAJIB isi § 0 Existing Analysis dulu (codebase audit + magnitude pick: NO-CHANGE/EXTEND/NEW-MODULE/REWRITE per Prinsip 19), baru drafting tech design + OpenAPI/proto + sequence diagram + 3 opsi + risk register. Human: Tech Lead pilih opsi, cross-tribe reviewer challenge magnitude.

Owner: Tech Lead Agent: Architect Gate 2: LGTM-TD + § 0 filled ~4-6 jam (vs. 2-3 hari)
3

Planning & Task Breakdown

Input: Tech design yang sudah signed. AI: Planner Agent generate impl plan: list task urut topologis, dependency graph, estimasi (T-shirt), parallelizable ya/tidak, risk flag. Human: Tech Lead validate breakdown, Dev claim task (auto-assigned di Jira).

Owner: Tech Lead + Dev Agent: Planner Gate 3: Plan signed ~1-2 jam
4

Development (+ Git Hygiene & UI-Impact Post-Dev v2.1)

Input: Task dari impl plan. Step 0 — Git Hygiene (Prinsip 20): cek working tree clean, git pull --ff-only, git checkout -b feat/<slug> di SETIAP repo target. AI: Coder Agent implement pakai CLAUDE.md + skill stack-specific. Wajib generate: code + unit test + evidence (curl.txt, test.log, screenshots/) + self-review checklist + isi UI-IMPACT.md Section 2 (kalau ada UI change) + invoke ui-impact-analyst Mode B untuk Section 3 (Agent Analysis). Human: Dev review hasil agent, iterate, open PR.

Owner: Dev Agent: Coder + Reviewer + UI-Impact-Analyst Gate 4: PR (branch feat/<slug>) + evidence ~40-60% lebih cepat
5

Testing

Input: PRD + tech design + acceptance criteria. AI: Tester Agent generate test plan (functional matrix, edge case, regression, perf hot-spot), drafting E2E test script (Playwright untuk Vue, integration test untuk Go). Human: QA review & supplement (yang AI bisa miss: domain quirks, regulasi). QA jalankan, agent bantu triase failure. Kunci: test plan disiapkan paralel dengan development, bukan setelah.

Owner: QA Agent: Tester Gate 5: Test plan green Bukan blocker lagi
6

Defects & QA Greenlight

Input: Bug report dari Gate 5. AI: Triage Agent classify (P0-P3, area, root-cause hypothesis), cek apakah ini lesson candidate. Human: Tech Lead & QA approve fix, tag lesson kalau pattern recurring.

Owner: QA + Tech Lead Agent: Triage Gate 6: QA sign-off
7

Live & Post-launch

Input: Build yang sudah QA-green. AI: Reliability Agent generate runbook, alert config, dashboard. Post-launch (T+7 hari) agent generate post-launch retrospective draft dari log + metrics + incident. Human: SRE / Tech Lead validate runbook, sign-off rilis. Setelah 7 hari, retro singkat → lesson baru.

Owner: SRE / Tech Lead Agent: Reliability Gate 7: Retro + lesson
Estimasi pemangkasan waktu: Grooming + Tech Disc yang biasanya 4-6 hari kalender → bisa jadi 1-1.5 hari kalender (dengan AI draft + 1-2 putaran review). Testing yang sering 3-5 hari blocker → paralel dengan development. Total: feature lead time turun 40-60% untuk feature medium.

4.2 Memory Federation — Lessons

Setiap tribe punya LESSONS.md sendiri. Tiap akhir bulan, Triage Agent run cross-tribe scan: lesson yang muncul di ≥2 tribe → promote ke org/.claude/LESSONS.md dan (kalau pattern engineering) dikodifikasi sebagai principle baru di org/.claude/CLAUDE.md.

SkenarioAksi
Tribe 3a temu pattern baru di paymentTulis di tribes/3/3a/LESSONS.md
Pattern muncul juga di 3b dan 3cPromote ke tribes/3/LESSONS.md + CLAUDE.md tribe 3
Tribe 1 & 2 juga punya kasus serupaPromote ke org/.claude/CLAUDE.md sebagai principle ke-19, 20, dst.

4.3 Cross-Tribe Review (Wajib di Gate 2)

Untuk tech design (Gate 2), minimal 1 reviewer dari tribe lain. Tujuan: bukan "permission" tapi cross-pollination — supaya pola di tribe 3 tidak buta tribe 1, dst. Reviewer punya 24 jam SLA; lewat itu, otomatis di-skip (tidak block).

Anti-pattern: Tribe yang membangun ulang CLAUDE.md dari nol. Selalu mulai dari org-level, override seperlunya. Kalau merasa >50% perlu di-override, itu sinyal arsitektur tribe terlalu beda dari org — bicarakan di forum tech lead, jangan fork diam-diam.

5. Feature Folder — Struktur Output Kanonik

5.3 Code Manifest — Kenapa Ada?

Code (Go handler/usecase/repo, Vue component) tetap harus di standard service layout — bukan di feature folder — karena Go module & bundler perlu layout standar. Tapi tetap harus bisa ditemukan dari feature folder.

Solusi: features/<slug>/G4-CODE-MANIFEST.md mencatat semua file code yang disentuh feature ini:

# Code Manifest — apple-pay-checkout

## Files added
- `internal/handler/http/apple_pay.go` (Gate 4 · Task T03)
- `internal/usecase/apple_pay_charge.go` (Gate 4 · Task T05)
- `internal/repository/apple_pay_token.go` (Gate 4 · Task T04)
- `web/src/components/ApplePayButton.vue` (Gate 4 · Task T07)

## Files modified
- `internal/handler/http/router.go` (route registration · Task T03)
- `api/openapi.yaml` (← di-generate dari `features/apple-pay-checkout/contract/openapi.yaml`)

## Migrations applied
- `migrations/2026_05_12_001_apple_pay_token.up.sql` (← copy dari feature folder)

## Standard locations updated (auto via `make feature-apply`)
- `monitoring/alerts/apple_pay_checkout.yaml`
- `runbooks/payment/apple-pay-checkout.md`

5.4 Build Tooling — make feature-apply

Beberapa artefak harus di-mirror ke standard location supaya tooling generic ikut. Ini di-handle satu perintah:

# Saat feature siap merge (Gate 4):
make feature-apply FEATURE=apple-pay-checkout
# Yang dilakukan:
# 1. Merge features/<slug>/contract/openapi.yaml → api/openapi.yaml (additive)
# 2. Copy features/<slug>/migrations/* → migrations/ (dengan timestamp)
# 3. Copy features/<slug>/runbook/alerts.yaml → monitoring/alerts/<slug>.yaml
# 4. Copy features/<slug>/runbook/failure-modes.md → runbooks/<service>/<slug>.md
# 5. Update G4-CODE-MANIFEST.md dengan timestamp

# Saat perlu rollback feature:
make feature-rollback FEATURE=apple-pay-checkout

Source of truth tetap di feature folder. Standard location adalah view yang di-regenerate dari feature folder.

5.5 Lifecycle Feature Folder

StatusAksiFolder location
ActiveDalam progres (Gate 1-7)features/<slug>/
Live & MaintainedSudah Gate 7 closed; folder tetap aktif sebagai referensifeatures/<slug>/
ArchivedFeature stabil ≥6 bulan; pindahkan ke arsip untuk de-clutterfeatures/_archived/<year>/<slug>/
DeprecatedFeature di-sunset; folder tetap untuk historical lessonfeatures/_archived/<year>/<slug>-DEPRECATED/
Manfaat konkret struktur ini:
  • Onboarding cepat: dev baru baca 1 folder, dia paham seluruh konteks feature
  • Audit trail: compliance/security audit cukup buka folder, semua bukti ada
  • Rollback bersih: kalau harus undo feature, semua artefak teridentifikasi di 1 tempat
  • Lesson extraction: retro ngeliat 1 folder, tahu apa yang berubah & kenapa
  • AI agent context: agent baca 1 folder = semua konteks feature; tidak perlu cari ke Slack/wiki
Anti-pattern yang dilarang:
  • PRD di Notion, tech design di Confluence, test plan di Google Doc — itu context drift
  • Migration di migrations/ root tanpa link ke feature mana yang melahirkannya
  • Alert config & runbook hilang setelah feature live
  • Evidence cuma di screenshot di chat PR

6. Tooling & Setup

Stack minimum yang dibutuhkan tiap dev. Tujuannya zero-config: clone repo → make agent-setup → siap kerja.

6.1 Claude Code Setup

Versi Claude Code terbaru. Per dev, sekali setup di shell:

npm install -g @anthropic-ai/claude-code   # atau sesuai instruksi resmi
# di root repo:
make agent-setup    # install hooks, link .claude/, validate CLAUDE.md, dll

6.2 Slash Commands (per fase)

Disimpan di org/.claude/commands/ dan otomatis tersedia di semua repo yang ikut struktur ini.

/tech-design <feature-slug>2Generate tech design + ADR + OpenAPI / proto + risk register
/plan <feature-slug>3Generate impl plan + task breakdown + estimate
/implement <task-id>4Implement task spesifik, generate code + test + evidence
/test-plan <feature-slug>5Generate test plan + skeleton E2E test

6.3 Skills — Stack-Specific Knowledge

Skill di-bundle di org/.claude/skills/. Per skill: SKILL.md + helper scripts + template kode.

  • golang-microservice — layout standar (cmd/, internal/, pkg/), pattern handler/service/repo, observability hook, graceful shutdown, error wrapping.
  • vue-frontend — composition API + Pinia store + script setup, 5 UI states (default/loading/empty/error/success), accessibility checklist.
  • grpc-contract — proto file convention, buf lint, backward-compat rule, codegen pipeline.
  • openapi-design — spec-first, validation matrix, 422 dengan friendly message (sesuai prinsip v1).
  • test-scaffolding — Go: table-driven test + testcontainers untuk integration. Vue: Vitest + Playwright.

6.4 MCP Servers (opsional, kuat)

Kalau tim siap, sambungkan MCP untuk integrasi langsung:

  • Jira/Linear MCP — Groomer Agent tarik epic, update status, comment
  • GitHub MCP — Coder Agent buka PR, baca diff, baca CI log
  • Confluence/Notion MCP — sync PRD/Tech Design ke wiki tim
  • Monitoring MCP (Grafana/Datadog) — Reliability Agent baca metric + log untuk post-launch retro
Saran rollout MCP: Mulai dari GitHub MCP saja (paling impactful untuk gate 4). Tambah Jira MCP di bulan 2. Tambah lainnya kalau benar-benar dibutuhkan — jangan kelebihan tool.

7. Templates & Prompts

Template fisik ada di folder templates/ di playbook ini. Ringkasan struktur tiap dokumen di bawah. Filosofi: setiap dokumen punya skeleton yang sama di semua tribe supaya reviewer dari tribe lain tidak perlu belajar format baru.

7.1 PRD Template

  • Problem statement (1 paragraf, no jargon)
  • Target user & jobs-to-be-done
  • In scope / Out of scope (eksplisit)
  • User stories + acceptance criteria (format Given/When/Then)
  • Non-functional requirements (perf, security, privacy)
  • Dependencies (service lain, tim lain, vendor)
  • Risks & mitigations
  • Open questions (auto-generated oleh Groomer; harus tuntas sebelum Gate 1)

7.2 Tech Design Template

  • Context (link ke PRD)
  • Options considered (minimal 3, dengan trade-off matrix)
  • Decision + rationale (= ADR inline)
  • API contract (OpenAPI snippet untuk REST, proto untuk gRPC)
  • Data model (ERD atau struct)
  • Sequence diagram (Mermaid, jangan PowerPoint)
  • Failure modes & recovery
  • Observability plan (apa yang di-log/metric/trace)
  • Rollout plan (feature flag? canary? big-bang?)
  • Cross-tribe reviewer (nama + tribe)

7.3 Test Plan Template

  • Scope (mapping ke acceptance criteria di PRD)
  • Test matrix (state × action; tiap baris = 1 test case)
  • Edge cases (Tester Agent generate; QA tambahkan domain-specific)
  • Automation strategy (mana unit, mana integration, mana E2E)
  • Regression set (tag test mana yang masuk regression suite)
  • Performance / load (jika relevan)
  • Exit criteria (pass rate, defect density threshold)

7.4 CLAUDE.md Template (Service-level)

  • What this service does (1-2 paragraf, no marketing)
  • Owners (primary + secondary on-call)
  • Stack & layout (Go versi, layout folder)
  • External dependencies (DB, queue, downstream service)
  • Local dev (cara run, test, debug — 1 perintah ideal)
  • Conventions yang nonstandard (kalau ada deviasi dari org)
  • Gotchas (hal yang sering nyangkut — dari LESSONS.md)

File konkret: lihat templates/ di folder ini.

8. Governance & Rollout

8.1 Rollout 90 Hari (rekomendasi)

FaseDurasiAksiSukses kriteria
Pilot (1 tribe) Minggu 1-4 Pilih 1 tribe (saran: tribe 3a — context kuat, 4-8 dev). Setup CLAUDE.md hierarchy, slash commands, 2-3 skill paling relevan. Jalankan 2-3 feature end-to-end. 1 feature lulus 7 gate. Lead time turun minimum 25%. Tim merasa "ini bantuan, bukan beban".
Replikasi (3 tribe) Minggu 5-8 Rollout ke tribe 3b, 3c (share context dengan 3a, paling murah). Mulai cross-tribe review. Aktifkan lesson promotion bulanan. Tribe 3 jalan paralel pakai playbook v2. Cross-tribe review jalan minimal 50% gate 2.
Skala (semua tribe) Minggu 9-12 Rollout tribe 1 & 2. Setup MCP Jira (atau equivalent). Bikin internal showcase / brownbag per bulan. Semua tribe pakai playbook. Lead time turun 40%+. Min. 5 lesson di-promote ke org-level.

8.2 Quick Start untuk Tribe Baru (1 hari)

  1. Pagi: Tech Lead clone playbook v2 → adapt tribes/X/CLAUDE.md (cukup isi domain glossary + 5 do/don't tribe). Setup repo skeleton.
  2. Siang: Walkthrough 1 jam ke seluruh dev tribe: 4 prinsip baru + 7 gate + slash command. Tunjukkan demo /groom + /tech-design di feature yang sedang antre.
  3. Sore: Tiap dev coba /implement di 1 task kecil. Catat friction. Friction → backlog improvement playbook.

8.3 Versi & Maintenance Playbook

  • Playbook ada di repo terpisah (atau org/.claude/). Versi semantik: major.minor. Major bump kalau workflow gate berubah; minor kalau template/command ditambah.
  • Quarterly review: 1 jam tiap kuartal — tech lead tribe + 1 PM + 1 QA. Bahas: lesson apa yang naik ke principle, command/skill apa yang ditambah/dibuang, gate mana yang sering di-skip (sinyal gate problem).
  • Owner playbook ditugaskan eksplisit (1 orang, rotate per kuartal — biar ownership tidak diffuse).

9. Metrik Sukses — Cek Realita, Bukan Vibe

4 metrik wajib, di-track tribe-level dan org-level. Auto-generated dari Jira + GitHub setiap minggu.

Lead time

Hari dari Gate 1 (PRD signed) → Gate 7 (live)

Target: turun 40% dalam 90 hari. Median, bukan rata-rata (outlier menipu).

Defect escape rate

Bug yang ketemu di produksi / total bug

Target: stagnan atau turun. Kalau naik = AI cuma mempercepat tapi kualitas turun. Stop dan investigasi.

Review time

Median jam dari PR open → merge

Target: turun 30%. Sinyal bahwa review fokus ke keputusan, bukan style.

Lesson velocity

Jumlah lesson baru per bulan

Target: ≥3 per tribe per bulan. Lesson terlalu sedikit = tim tidak refleksi; terlalu banyak = ada masalah sistemik.

Hati-hati metric gaming. Kalau lead time turun tapi defect escape naik, itu bukan win. Selalu lihat keempat metrik bersamaan.

10. Anti-pattern Khusus Tim

  1. "AI bilang begitu" sebagai justifikasi. Output AI tetap perlu reasoning manusia. Approval gate ada karena ini.
  2. Skip gate karena buru-buru. Skip 1 gate = utang yang dibayar 3× di gate berikutnya. Kalau benar-benar urgent, ada emergency lane yang punya retro wajib.
  3. Tribe fork CLAUDE.md. Kalau merasa override >50%, hentikan — bahas di forum tech lead.
  4. Reviewer cross-tribe perfunctory. Reviewer cross-tribe minimal harus tulis 1 pertanyaan / saran spesifik; "LGTM" kosong tidak diterima.
  5. Slash command jadi black box. Tiap command harus punya doc 1 halaman: input, output, asumsi. Kalau tidak, hilangkan.
  6. Lesson loop mati. Kalau 1 bulan tidak ada lesson naik, owner playbook wajib facilitate retro mini.
  7. Approval pakai chat. Approval di Slack tidak terhitung. Harus di artefak yang bisa di-grep nanti.
  8. "Ini buat developer senior aja". Justru junior paling diuntungkan playbook. Pastikan onboarding masuk hari ke-2 dev baru.
