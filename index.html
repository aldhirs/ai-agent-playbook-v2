Evolusi: v1 → v2 → v3

Playbook ini sudah lewat 2 major iteration. Setiap iterasi address pain dari iterasi sebelumnya tanpa membuang yang sudah jalan.

1. Prinsip Operating + Quality (15-20)

v1 punya 14 prinsip teknis (contract first, no over-claim, validation by default, dll — retain di v3). v2 tambah 6 prinsip operating + quality untuk tim — semua tetap apply di v3:

Prinsip 15

AI Proposes, Human Approves

Setiap gate ditutup oleh artefak yang di-approve oleh peran manusia tertentu. Approval = signed comment di PR / tag di doc, bukan verbal. Tanpa approval, gate tidak boleh maju.

Prinsip 16

Tribe Shares, Service Owns

Pola yang dipakai >1 service di tribe yang sama wajib naik ke tribes/X/CLAUDE.md. Pola yang dipakai >1 tribe wajib naik ke org/CLAUDE.md. Konvensi diturunkan, bukan ditemukan ulang.

Prinsip 17

One Folder, All Phases

Setiap feature/epic punya satu folder features/<slug>/ yang berisi seluruh artefak dari grooming sampai post-launch. Tidak ada konteks yang hidup di Slack/DM/kepala orang.

Prinsip 18

Async by Default, Sync on Conflict

Agent menghasilkan draft async (jalan di background). Manusia review batched (pagi/sore). Sync meeting hanya dipanggil saat ada konflik desain, bukan untuk ratifikasi yang AI sudah selesai.

Prinsip 19 ⭐ Quality

Analyze Existing First, Propose Minimal Change

Sebelum design solusi baru, agent WAJIB audit codebase existing. Tentukan magnitude: NO-CHANGE / EXTEND (default) / NEW-MODULE / REWRITE. Naik harus justify dengan ADR.

Prinsip 20 ⭐ Quality

Sync Repo & Branch dari Slug

Agent yang akan modify file di repo target WAJIB git hygiene 3-step: (1) cek working tree clean, (2) git pull --ff-only di base branch, (3) git checkout -b feat/<slug>. Berlaku per-repo.

Kenapa 6 prinsip ini cukup? Mereka menutup 6 mode kegagalan tim: keputusan ambigu (15), reinventing the wheel (16), context drift (17), rapat tak perlu (18), over-engineering (19), dan stale code / branch chaos (20).

3. Workflow 3 Phase — Dari Ide ke Live (v3.x)

v3 collapse 7-gate granular jadi 3 phase dengan 1 checkpoint pra-dev. Phase 1 autonomous (AI generate consolidated spec), Phase 2 paralel BE/FE dev, Phase 3 retain Gate 5-7 v2.x (QA + SRE owner).

Command sequence (happy path): 
/spec <epic>                    → Phase 1 autonomous (3 agent sequential + IMPL-PLAN aggregator)
LGTM-SPEC-BE + LGTM-SPEC-FE     → Phase 1.5 checkpoint (grep-able di SPEC § 12)
/implement <slug>/be (+ /fe)    → Phase 2 dev (paralel via [BE-CONTRACT-FROZEN])
qa-greenlight + sre runbook     → Phase 3 (retain v2.x Gate 5-7)
1

Spec (autonomous) v3.0

Input: Epic mentah (Jira/Linear link atau deskripsi 1-3 paragraf). AI: /spec command invoke 3 agent sequential — (1) pm fill SPEC-BE § 1-2 + SPEC-FE § 1, (2) solution-architect fill SPEC-BE § 0, 3-11 + MARK 🔒 [BE-CONTRACT-FROZEN] di § 4, (3) ui-impact-analyst fill SPEC-FE § 0, 2-11 + generate wireframes/*.html (Tailwind CDN interactive 5-state). v3.1: Step 4 aggregator generate IMPL-PLAN.md dengan timeline + critical path ASCII gantt.

Push-back protocol: kalau ada ambiguity, agent tulis ke OPEN-QUESTIONS.md dengan format pertanyaan + why-blocking + default-assumption. Cap 2 round, round 3 escalate ke meeting.

Owner: PM (consulted) Agent: pm + solution-architect + ui-impact-analyst Output: 2-3 file SPEC + wireframes + contract
1.5

Checkpoint — Single Human Approval (split BE+FE)

Bukan 3 approval terpisah seperti v2 (LGTM-PRD, LGTM-TD, LGTM-PLAN) — sekarang cuma 1 checkpoint dengan 2 LGTM paralel:

  • BE Dev + TL review SPEC-BE.md → tulis LGTM-SPEC-BE by <nama> on YYYY-MM-DD di § 12
  • Designer + FE Dev review SPEC-FE.md + buka wireframes/index.html di browser → tulis LGTM-SPEC-FE di § 12
  • Optional: PM + TL review IMPL-PLAN.md untuk adjust target sprint → tulis LGTM-PLAN di § 7
Owner: BE Dev + TL · Designer + FE Dev Approval: LGTM-SPEC-BE + LGTM-SPEC-FE (grep-able)
2

Dev (paralel BE+FE)

Input: SPEC dengan LGTM. Step 0 — Git Hygiene (Prinsip 20): cek working tree clean, git pull --ff-only, git checkout -b feat/<slug>-<chunk>. AI: 2 agent invoke paralel — backend-engineer via /implement <slug>/be + frontend-engineer via /implement <slug>/fe. Saat [BE-CONTRACT-FROZEN] marker ada, FE Dev sah mulai paralel work (mock API berdasarkan SPEC-BE § 4, switch ke real saat BE PR merged). Output per chunk: 1 PR (soft cap 500 lines) + 3 artefak evidence (curl.txt, test.log, screenshots/) + post-dev wireframe verification via ui-impact-analyst Mode B.

Owner: BE Dev + FE Dev Agent: backend-engineer + frontend-engineer Exit: peer review + CI hijau + evidence + make verify-feature exit 0
3

Test & Live (retain Gate 5-7 v2.x)

Phase 3 tidak collapse — Gate 5 (Testing), Gate 6 (Defects & Greenlight), Gate 7 (Live & Post-launch) tetap pakai workflow v2.x dengan QA + SRE sebagai owner. Reasoning: stakeholder beda, lifecycle beda, value collapse rendah.

  • Gate 5 Testing: qa-engineer via /test-plan → AC mapping + 5-state matrix + 10+ edge case → qa-greenlight
  • Gate 6 Defects: defect classification + root-cause + lesson candidate → defect closed + lesson linked
  • Gate 7 Live: sre generate runbook + alert + dashboard. T+7 retro post-launch → lesson baru
Owner: QA · SRE / TL Agent: qa-engineer + sre Approval: qa-greenlight + SRE checklist + retro committed
Estimasi pemangkasan waktu (v2 → v3): Pra-dev (Grooming + Tech Disc + Planning) yang v2 perlu 4-6 hari + 3 approval → v3 jadi ~1 hari + 1 checkpoint (autonomous spec generation + paralel BE/FE review). FE handoff suffer (Tanisa Konsul case) diaddress via wireframe HTML preview. BE-FE paralel work via [BE-CONTRACT-FROZEN] save 30-50% wall-clock di Phase 2. Total feature lead time: turun 60-70% untuk feature medium dibanding v2.

4.2 Memory Federation — Lessons

Setiap tribe punya LESSONS.md sendiri. Tiap akhir bulan, qa-engineer (triage mode) run cross-tribe scan: lesson yang muncul di ≥2 tribe → promote ke org/.claude/LESSONS.md dan (kalau pattern engineering) dikodifikasi sebagai principle baru di org/.claude/CLAUDE.md.

SkenarioAksi
Tribe 3a temu pattern baru di paymentTulis di tribes/3/3a/LESSONS.md
Pattern muncul juga di 3b dan 3cPromote ke tribes/3/LESSONS.md + CLAUDE.md tribe 3
Tribe 1 & 2 juga punya kasus serupaPromote ke org/.claude/CLAUDE.md sebagai principle ke-19, 20, dst.

4.3 Cross-Tribe Review (Wajib di Phase 1.5 Checkpoint)

Untuk SPEC-BE (Phase 1.5 checkpoint), minimal 1 reviewer dari tribe lain di-tag sebagai LGTM-SPEC-BE consultant. Tujuan: bukan "permission" tapi cross-pollination — supaya pola di tribe 3 tidak buta tribe 1, dst. Reviewer punya 24 jam SLA; lewat itu, otomatis di-skip (tidak block).

Anti-pattern: Tribe yang membangun ulang CLAUDE.md dari nol. Selalu mulai dari org-level, override seperlunya. Kalau merasa >50% perlu di-override, itu sinyal arsitektur tribe terlalu beda dari org — bicarakan di forum tech lead, jangan fork diam-diam.

5.3 Code Tracking — PR Description & SPEC Reference (v3.x)

Code (Go handler/usecase/repo, Vue component) tetap di standard service layout. Tracking "file apa saja yang disentuh" sekarang di PR description (link ke SPEC-BE/SPEC-FE section per perubahan), bukan separate G4-CODE-MANIFEST.md file (deprecated di v3).

Pattern PR description (v3.x):

## SPEC reference
- SPEC-BE.md § 3 (Migration) → migrations/2026_05_25_001_apple_pay_token.up.sql
- SPEC-BE.md § 4 (API Contract) → internal/handler/http/apple_pay.go
- SPEC-BE.md § 5 (Usecase) → internal/usecase/apple_pay_charge.go

## Self-review checklist (refer commands/implement.md § 7)
- [x] Git hygiene done (Prinsip 20)
- [x] Contract conformance (match SPEC-BE § 4)
- [x] § 0 Reuse plan diikuti
- [x] Observability lengkap (log + metric + trace per § 6)
- [x] Banned-field test green
- [x] PR diff <500 lines (soft cap)

## Evidence
- features/apple-pay-checkout/evidence/be/curl.txt
- features/apple-pay-checkout/evidence/be/test.log

Kenapa deprecate G4-CODE-MANIFEST.md? Di v2.x ini berguna karena task breakdown granular (banyak PR per feature). Di v3.x cuma 2 PR per feature (BE + FE), jadi PR description sudah cukup sebagai single source of truth. Audit trail cross-PR tetap ada via SPEC reference section.

5.4 Build Tooling — make feature-apply

Beberapa artefak harus di-mirror ke standard location supaya tooling generic ikut. Ini di-handle satu perintah:

# Saat feature siap merge (Phase 2 — sebelum/saat PR open):
make feature-apply FEATURE=apple-pay-checkout
# Yang dilakukan:
# 1. Merge features/<slug>/contract/openapi.yaml → api/openapi.yaml (additive)
# 2. Copy features/<slug>/migrations/* → migrations/ (dengan timestamp)
# 3. Copy features/<slug>/runbook/alerts.yaml → monitoring/alerts/<slug>.yaml (Phase 3)
# 4. Copy features/<slug>/runbook/failure-modes.md → runbooks/<service>/<slug>.md (Phase 3)

# Saat perlu rollback feature:
make feature-rollback FEATURE=apple-pay-checkout

Source of truth tetap di feature folder. Standard location adalah view yang di-regenerate dari feature folder.

5.5 Lifecycle Feature Folder

StatusAksiFolder location
ActiveDalam progres (Phase 1 → Phase 3)features/<slug>/
Live & MaintainedSudah Gate 7 closed; folder tetap aktif sebagai referensifeatures/<slug>/
ArchivedFeature stabil ≥6 bulan; pindahkan ke arsip untuk de-clutterfeatures/_archived/<year>/<slug>/

Manfaat konkret struktur ini:
  • Onboarding cepat: dev baru baca 1 folder, dia paham seluruh konteks feature
  • Audit trail: compliance/security audit cukup buka folder, semua bukti ada
  • Rollback bersih: kalau harus undo feature, semua artefak teridentifikasi di 1 tempat
  • Lesson extraction: retro ngeliat 1 folder, tahu apa yang berubah & kenapa
  • AI agent context: agent baca 1 folder = semua konteks feature; tidak perlu cari ke Slack/wiki
Anti-pattern yang dilarang:
  • PRD di Notion, tech design di Confluence, test plan di Google Doc — itu context drift
  • Migration di migrations/ root tanpa link ke feature mana yang melahirkannya
  • Alert config & runbook hilang setelah feature live
  • Evidence cuma di screenshot di chat PR

6. Tooling & Setup

Stack minimum yang dibutuhkan tiap dev. Tujuannya zero-config: clone repo → make agent-setup → siap kerja.

6.1 Claude Code Setup

Versi Claude Code terbaru. Per dev, sekali setup di shell:

npm install -g @anthropic-ai/claude-code   # atau sesuai instruksi resmi

6.3 Subagents — Project-level Specialization

Playbook menyediakan 11 subagent file di agents/ (project-level, shareable via git, menang vs user-level). Refer ke §2 Operating Model untuk mapping per Phase.

  • Orchestration (Phase 1 via /spec): pm, solution-architect, ui-impact-analyst
  • Implementation (Phase 2 via /implement): backend-engineer (Go), frontend-engineer (Vue 3 / Nuxt) — customize per stack project
  • Quality (Phase 3 + cross-cutting): qa-engineer, security-engineer, sre, verification-gatekeeper
  • Support (on-demand): product-designer (UX principles), domain-expert-elearning-payment (worked example — copy + rename + adapt ke domain Anda)

Genericization: 4 agent (backend-engineer, frontend-engineer, solution-architect, ui-impact-analyst) pakai placeholder <your-shared-lib>, <your-fe-monorepo>, <your-app> — replace dengan path actual project saat adopt.

6.4 MCP Servers (opsional, kuat)

Kalau tim siap, sambungkan MCP untuk integrasi langsung:

  • Jira/Linear MCP — /spec tarik epic, update status, comment
  • GitHub/GitLab MCP — /implement buka PR, baca diff, baca CI log
  • Confluence/Notion MCP — sync SPEC-BE/SPEC-FE ke wiki tim
  • Monitoring MCP (Grafana/Datadog) — Phase 3 SRE baca metric + log untuk post-launch retro
  • Figma MCP — ui-impact-analyst fetch design context untuk Pre-Dev wireframe generation
Saran rollout MCP: Mulai dari GitHub MCP saja (paling impactful untuk gate 4). Tambah Jira MCP di bulan 2. Tambah lainnya kalau benar-benar dibutuhkan — jangan kelebihan tool.

8. Governance & Rollout

8.1 Rollout 90 Hari (rekomendasi)

FaseDurasiAksiSukses kriteria
Pilot (1 tribe) Minggu 1-4 Pilih 1 tribe (saran: tribe 3a — context kuat, 4-8 dev). Setup CLAUDE.md hierarchy, slash commands, 2-3 skill paling relevan. Jalankan 2-3 feature end-to-end. 1 feature lulus 7 gate. Lead time turun minimum 25%. Tim merasa "ini bantuan, bukan beban".
Replikasi (3 tribe) Minggu 5-8 Rollout ke tribe 3b, 3c (share context dengan 3a, paling murah). Mulai cross-tribe review. Aktifkan lesson promotion bulanan. Tribe 3 jalan paralel pakai playbook v2. Cross-tribe review jalan minimal 50% gate 2.
Skala (semua tribe) Minggu 9-12 Rollout tribe 1 & 2. Setup MCP Jira (atau equivalent). Bikin internal showcase / brownbag per bulan. Semua tribe pakai playbook. Lead time turun 40%+. Min. 5 lesson di-promote ke org-level.

8.2 Quick Start untuk Tribe Baru (1 hari)

  1. Pagi: Tech Lead clone playbook v2 → adapt tribes/X/CLAUDE.md (cukup isi domain glossary + 5 do/don't tribe). Setup repo skeleton.
  2. Siang: Walkthrough 1 jam ke seluruh dev tribe: 4 prinsip baru + 7 gate + slash command. Tunjukkan demo /groom + /tech-design di feature yang sedang antre.
  3. Sore: Tiap dev coba /implement di 1 task kecil. Catat friction. Friction → backlog improvement playbook.

8.3 Versi & Maintenance Playbook

  • Playbook ada di repo terpisah (atau org/.claude/). Versi semantik: major.minor. Major bump kalau workflow gate berubah; minor kalau template/command ditambah.
  • Quarterly review: 1 jam tiap kuartal — tech lead tribe + 1 PM + 1 QA. Bahas: lesson apa yang naik ke principle, command/skill apa yang ditambah/dibuang, gate mana yang sering di-skip (sinyal gate problem).
  • Owner playbook ditugaskan eksplisit (1 orang, rotate per kuartal — biar ownership tidak diffuse).

9. Metrik Sukses — Cek Realita, Bukan Vibe

4 metrik wajib, di-track tribe-level dan org-level. Auto-generated dari Jira + GitHub setiap minggu.

Lead time

Hari dari Phase 1.5 (LGTM-SPEC signed) → Phase 3 (live + retro committed)

Target v3: turun 60-70% vs v2 baseline. Median, bukan rata-rata (outlier menipu).

Defect escape rate

Bug yang ketemu di produksi / total bug

Target: stagnan atau turun. Kalau naik = AI cuma mempercepat tapi kualitas turun. Stop dan investigasi.

Review time

Median jam dari PR open → merge

Target: turun 30%. Sinyal bahwa review fokus ke keputusan, bukan style.

Lesson velocity

Jumlah lesson baru per bulan

Target: ≥3 per tribe per bulan. Lesson terlalu sedikit = tim tidak refleksi; terlalu banyak = ada masalah sistemik.

Hati-hati metric gaming. Kalau lead time turun tapi defect escape naik, itu bukan win. Selalu lihat keempat metrik bersamaan.

10. Top 5 Anti-pattern (yang paling sering bikin tim macet)

Daftar di-curate dari retro lintas-tribe. Lainnya ada di Gate 6 Defects & Greenlight.

#1 · Process failure

Approval di chat

Slack/DM/verbal tidak terhitung. Approval HARUS di artefak yang bisa di-grep nanti — comment LGTM-PRD di file PRD, bukan thumbs-up di chat.

#2 · Justification failure

"AI bilang begitu"

Output AI tetap butuh reasoning manusia. Approval gate ada karena ini. Reviewer wajib bisa jelaskan kenapa decision, bukan "agent yang bilang".

#3 · Velocity trap

Skip gate karena buru-buru

Skip 1 gate = utang dibayar 3× di gate berikutnya. Urgent banget? Ada emergency lane dengan retro wajib ≤72 jam — bukan ban gate.

#4 · Review failure

Cross-tribe reviewer "LGTM" kosong

Reviewer cross-tribe WAJIB tulis minimal 1 pertanyaan / saran spesifik. "LGTM" tanpa konteks ditolak — kalau tidak ada saran, dia belum baca.

#5 · Quality failure (v2.1)

Skip § 0 Existing Analysis di Phase 1 Spec

Loncat ke "3 options brainstorm" tanpa audit codebase = over-engineer. Default magnitude EXTEND, naik harus justify dengan ADR. Section 0 kosong = tech design ditolak.

#6 · Workflow failure (v2.1)

Edit langsung di main tanpa branch

Bypass code review + dirty git history. Prinsip 20: WAJIB git checkout -b feat/<slug> sebelum sentuh code. Branch naming ad-hoc juga ditolak.

Anti-pattern lain (3 item lagi yang lebih jarang) →
  1. Tribe fork CLAUDE.md. Override >50% = sinyal arsitektur tribe terlalu beda dari org. Bahas di forum tech lead, jangan fork diam-diam.
  2. Slash command jadi black box. Tiap command WAJIB punya doc 1 halaman: input, output, asumsi. Tanpa doc = hilangkan.
  3. Lesson loop mati. 1 bulan tanpa lesson naik = sinyal proses macet. Owner playbook facilitate retro mini.
